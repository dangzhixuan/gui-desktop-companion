# 第一阶段：骨层修正说明

这一阶段的目标不是增加可见功能，而是让数据、安全和业务状态足够可靠，
使后续 GUI、定时器和 AI 分析建立在稳定基础上。

## 1. 配置与秘密分离

普通配置（人格、时间、模型名）可以保存在 `config.toml`。
API Key 属于秘密，只从环境变量 `DEEPSEEK_API_KEY` 读取。

Windows PowerShell 临时设置（仅当前终端有效）：

```powershell
$env:DEEPSEEK_API_KEY = "你的新密钥"
python demo_greet.py
```

长期设置可使用 Windows“编辑系统环境变量”界面。不要把真实密钥写进
`.env.example`、`config.toml`、代码或简历项目仓库。

已经公开过的密钥必须在服务商后台撤销；仅从代码中删除并不能使旧密钥失效。

## 2. 当前状态与历史快照分离

`tasks` 保存任务当前状态：

- `pending`：未完成
- `done`：已完成，并记录 `completed_at`
- `dropped`：已放弃，并记录 `dropped_at`

`task_reviews` 保存每日复盘快照：

- 哪一天复盘；
- 当时确认的状态；
- 未完成或放弃的原因；
- 实际复盘时间。

同一天、同一任务只有一条复盘记录，通过唯一约束和 UPSERT 保证重复提交不会
制造重复数据。

这种设计让系统既能回答“任务现在怎样”，也能回答“过去某天为什么没完成”。

## 3. 数据库迁移

SQLite 的 `PRAGMA user_version` 保存数据库结构版本。当前版本为 2：

- v1：`tasks` 与 `daily_logs`
- v2：`tasks.dropped_at` 与 `task_reviews`

程序启动时按版本顺序升级旧库，不要求用户删除数据库。迁移在事务中执行，
失败时不会留下只升级一半的结构。

## 4. 为什么需要专用复盘查询

“今日待办”和“今日复盘”不是同一集合：

- 今日待办只看仍未完成的任务；
- 今日复盘还必须包含今天已经完成或放弃的任务。

因此新增 `get_tasks_for_review()`，避免白天完成任务后，晚上反而看不到它。

## 5. AI 输出不可信

LLM 即使被要求返回 JSON，也可能返回普通文本、错误类型或不合规列表。
`_normalize_review_result()` 把模型输出校验为固定结构：

```python
{"comment": "字符串", "plan": ["字符串", "..."]}
```

这属于 AI 产品中的“确定性外壳”：模型可以概率化生成，但进入数据库和 UI 前，
必须经过传统程序校验。

## 6. 自动化测试

运行：

```powershell
python -m unittest discover -s tests -v
```

测试覆盖：

- 环境变量密钥读取；
- 配置时间校验；
- 旧数据库无损迁移；
- 已完成任务仍进入当晚复盘；
- 放弃时间与原因持久化；
- 复盘 UPSERT 防重复；
- 晚间闭环写回明日规划；
- 异常模型输出降级。

这批测试是需求的可执行验收标准。后续重构 GUI 或调度器时，只要它们继续通过，
就说明核心数据闭环没有被破坏。
